# UX 情報充足度 評価チェックリスト

**このファイルは Naoya さん専用の判断シートです。**
**Cursor が MVP UI 実装を完了したら、11 件のデータを実機で触りながら、以下のチェックリストで情報充足度を判断してください。**

---

## 評価の目的

11 件のサンプルデータで **UI の受け皿が妥当か** を確認し、以下を決めます:

1. 現在のスキーマ(v1.1.0)のまま A2 collocation の残り約 300 件を量産してよいか
2. スキーマ改定が必要な場合、どのフィールドをどう変更すべきか
3. データ生成プロンプト(Chat 生成テンプレート)の改訂が必要か

判断結果を持って設計チャット(現在の会話 or それに準じる Claude チャット)に戻り、次の打ち手を決めます。

---

## 評価環境の準備

- MacBook(A 端末): Chrome or Safari で `pnpm dev` した localhost にアクセス
- iPhone(C 端末): 同じ URL にアクセス(同一 Wi-Fi 経由 or GitHub Pages 本番デプロイ)
- 各端末で 11 件全てを一通り触ること
- **静かな環境で 30 分〜1 時間** を確保。急いで触ると判断が甘くなります

---

## 評価パート 2: Mode A(識別)(10 分)

### 2-1. distractor 方式の比較

**同じ item を、両方の方式で少なくとも 5 回ずつ試す。**

#### 方式 1: ランダム
- [ ] 4 択が **易しすぎる**(誤答が明らかに違う分野)
- [ ] 4 択が **適度に難しい**
- [ ] 4 択が **難しすぎる**(判別不能)

#### 方式 2: confusables 優先
- [ ] 4 択が **易しすぎる**
- [ ] 4 択が **適度に難しい**(混同の勉強になる)
- [ ] 4 択が **難しすぎる**(区別できない)

**判定**: Mode A の distractor は
- □ ランダム推奨(confusables は情報として惜しいが distractor には不向き)
- □ confusables 推奨(学習効果高い)
- □ ハイブリッド推奨(confusables 3 件未満のときのみランダム補充)

### 2-2. 選択後のフィードバック

- [ ] 正解時の表示は十分か
- [ ] 誤答時に「なぜ違うか」の説明があった方がよいか
  - □ 現状で十分(判定のみ)
  - □ confusables の `key_difference_ja` を表示すべき
- [ ] 「詳細を見る」ボタンの導線は自然か

### 2-3. 出題順の妥当性

- [ ] 11 件のみだと同じ item がすぐ再出題される問題があるか
  - □ 気にならない(11 件で回して OK)
  - □ 同じ item をより長く避ける仕組みが必要(実装済みは連続のみ回避)

## 評価パート 3: Mode B(想起)(10 分)

### 3-1. ヒントレベルの比較

**3 つのレベルを 5 回ずつ試す。**

- [ ] レベル 1(訳のみ): 思い出すのに **十分 / 難しすぎる / 易しすぎる**
- [ ] レベル 2(訳 + 定義): 思い出すのに **十分 / 難しすぎる / 易しすぎる**
- [ ] レベル 3(訳 + 定義 + semantic_field): 思い出すのに **十分 / 難しすぎる / 易しすぎる**

**判定**: デフォルトのヒントレベルは
- □ レベル 1
- □ レベル 2
- □ レベル 3
- □ ユーザーが選択できるままがよい

### 3-2. 解答表示の情報量

- [ ] 解答画面に **surface / IPA / 例文 1 つ** で十分か
- [ ] 追加すべき情報:
  - □ definition_en(意味の確認)
  - □ common_errors_ja の 1 つ(自分の思い出し方が間違っていないか確認)
  - □ その他: __________

### 3-3. 自己申告 vs 入力式

MVP は自己申告(思い出せたかどうかは自分で判断)ですが、Phase 2 本実装で入力式にする価値があるか:

- [ ] 自己申告で十分(実際に思い出す訓練になる)
- [ ] 入力式が必要(甘え防止のため)
- [ ] 選択できる方がよい

## 次のアクションフロー

判定 A の場合:
→ 設計チャットに戻り「量産 GO」と報告  
→ `chat-generation-template-a2.md` を使って新チャットで A2 collocation 量産開始

判定 B / C / D の場合:
→ 設計チャットに戻り、上記チェックボックスの記入内容を報告  
→ スキーマ改定 or プロンプト改訂の指示書を作成  
→ 場合により再度パイロットで検証してから量産開始

判定 E の場合:
→ 追加パイロット(8 件)の指示書を作成  
→ 得られた結果で再度本チェックリストを実施

## 補足: 各判定に伴う手戻りコスト

判断を先送りにすると、後の手戻りコストが大きくなります。参考として:

| 段階 | 手戻りコスト | 影響範囲 |
|---|---|---|
| **今の段階(11件)** | 小 | 11 件のデータ再生成 + UI 調整のみ |
| A2 collocation 完了後(300件) | 中 | 300 件を再生成、UI 大改修の可能性 |
| A2 全体完了後(2,430件) | 大 | 2,430 件再生成、モデル呼び出しコスト膨大 |
| B1 着手後 | 極大 | 累積したデータをすべて手直し |

したがって **今の段階でしっかり判断すること** が最も費用対効果が高いです。多少時間をかけて 30 分〜1 時間の評価を行うことを推奨します。
